# 2페이지. Stack Parameters와 Stack Frame

## 1. Stack frame이란?

**Stack frame**은 procedure 하나가 실행되는 동안 stack 위에 만들어지는 작업 공간이다.

procedure가 실행되려면 다음과 같은 정보들이 필요하다.

```text
1. Passed arguments
   호출자가 procedure에게 전달한 값 또는 주소

2. Subroutine return address
   procedure가 끝났을 때 돌아갈 코드 위치

3. Local variables
   procedure 내부에서만 사용하는 지역 변수

4. Saved registers
   procedure가 사용하기 전에 저장해둔 register 값
```

C 함수로 보면 다음과 같다.

```c
int AddTwo(int x, int y)
{
    int sum;
    sum = x + y;
    return sum;
}
```

이 함수가 실행되려면 stack에 최소한 다음 정보가 필요하다.

```text
x, y              // parameter
return address    // 함수 끝나고 돌아갈 위치
old EBP           // 이전 stack frame 복구용
sum               // local variable
```

---

## 2. Stack frame 생성 순서

강의자료는 stack frame이 만들어지는 과정을 6단계로 설명한다.

### Step 1. Passed arguments가 stack에 push됨

procedure 호출 전에 caller가 argument를 stack에 push한다.

예를 들어 C에서 다음과 같이 호출한다고 하자.

```c
AddTwo(5, 6);
```

assembly에서는 보통 다음과 같은 형태가 된다.

```asm
push 6
push 5
call AddTwo
```

C/C++ calling convention에서는 argument를 보통 **오른쪽에서 왼쪽 순서**, 즉 reverse order로 push한다.

---

### Step 2. CALL이 return address를 push함

```asm
call AddTwo
```

이 명령어가 실행되면 CPU는 자동으로 **return address**를 stack에 push한다.

return address는 procedure가 끝난 뒤 다시 실행해야 하는 다음 명령어의 주소이다.

```asm
call AddTwo
; AddTwo가 끝나면 여기로 돌아와야 함
```

따라서 `RET` 명령어는 stack에서 return address를 꺼내 EIP에 넣는다. 그 결과 프로그램 실행 흐름이 caller로 돌아간다.

---

### Step 3. Procedure 시작 시 EBP를 저장함

procedure는 보통 다음과 같이 시작한다.

```asm
push ebp
mov  ebp, esp
```

먼저 기존 EBP를 stack에 저장한다.

```asm
push ebp
```

왜냐하면 caller도 자기 stack frame을 EBP 기준으로 관리하고 있었을 수 있기 때문이다. callee가 EBP를 그냥 바꿔버리면 caller의 기준점이 사라진다.

그래서 기존 EBP를 저장하고, procedure가 끝날 때 다시 복구한다.

---

### Step 4. EBP = ESP

```asm
mov ebp, esp
```

이제 EBP는 현재 procedure의 stack frame 기준점이 된다.

이 시점부터 parameter와 local variable은 EBP를 기준으로 접근한다.

대표적인 구조는 다음과 같다.

```text
높은 주소
────────────────────────
[EBP+12]   second parameter
[EBP+8]    first parameter
[EBP+4]    return address
[EBP]      saved old EBP
[EBP-4]    local variable 1
[EBP-8]    local variable 2
────────────────────────
낮은 주소
```

이 구조는 이 장 전체에서 계속 반복된다.

---

### Step 5. Local variable 공간 확보

local variable이 있으면 ESP를 감소시켜 stack 공간을 확보한다.

```asm
sub esp, 8
```

이 코드는 stack에 8바이트 공간을 만드는 것이다. 예를 들어 DWORD local variable 2개를 만들 수 있다.

왜 ESP를 감소시키는가?

x86 stack은 보통 **높은 주소에서 낮은 주소 방향으로 성장**하기 때문이다.

---

### Step 6. 필요한 register를 저장함

procedure 내부에서 register를 사용할 때 caller의 register 값을 보존해야 한다면, 해당 register를 stack에 push해서 저장한다.

```asm
push ebx
push esi
...
pop esi
pop ebx
```

이때 복구 순서는 push의 반대가 되어야 한다.

```asm
push ebx
push esi
```

했다면,

```asm
pop esi
pop ebx
```

해야 한다. stack은 LIFO 구조이기 때문이다.

---

## 3. 이 페이지의 핵심

```text
Stack frame은 procedure 하나가 실행되는 동안 stack에 만들어지는 작업 공간이다.
EBP는 stack frame의 기준점 역할을 한다.
parameter는 보통 [EBP+8], [EBP+12]처럼 EBP보다 위쪽에 있다.
local variable은 보통 [EBP-4], [EBP-8]처럼 EBP보다 아래쪽에 있다.
```

# 3페이지. Disadvantages of Register Parameters

이 페이지는 **register parameter 방식의 단점**을 설명한다.

## 1. Register parameter란?

Register parameter 방식은 argument를 stack에 push하지 않고 register에 넣어서 전달하는 방식이다.

예를 들어 `DumpMem` procedure를 호출한다고 하자.

```asm
mov esi, OFFSET array      ; array 시작 주소
mov ecx, LENGTHOF array    ; array 길이
mov ebx, TYPE array        ; 원소 크기
call DumpMem
```

이 방식은 빠를 수 있다. register 접근은 memory 접근보다 훨씬 빠르기 때문이다.

Microsoft의 `fastcall` convention도 이런 아이디어를 사용한다. parameter 일부를 register로 넘겨서 runtime efficiency를 높이려는 것이다.

---

## 2. 문제: register는 이미 다른 용도로 쓰인다

register는 parameter 전달에만 쓰이는 공간이 아니다.

예를 들어 caller가 다음과 같이 중요한 값을 register에 저장해두었다고 하자.

```asm
mov ebx, someImportantValue
mov ecx, loopCount
mov esi, sourceAddress
```

그런데 procedure 호출을 위해 이 register들을 parameter 전달용으로 덮어쓰면 기존 값이 사라진다.

그래서 호출 전에 register 값을 저장해야 한다.

```asm
push ebx
push ecx
push esi

mov esi, OFFSET array
mov ecx, LENGTHOF array
mov ebx, TYPE array
call DumpMem

pop esi
pop ecx
pop ebx
```

---

## 3. 성능상 이점이 줄어듦

register parameter를 쓰는 이유는 빠르기 위해서이다.

그런데 register 값을 보존하려고 `push`, `pop`을 계속 해야 한다면, 추가 비용이 생긴다.

```text
register parameter의 의도:
    memory access 줄이고 빠르게 전달

실제 문제:
    register 보존을 위해 push/pop 필요
    코드 길이 증가
    실행 overhead 증가
```

따라서 register parameter의 성능상 장점이 줄어들 수 있다.

---

## 4. PUSH/POP 짝 맞추기 문제

register를 저장하고 복구할 때는 push/pop 짝이 정확해야 한다.

```asm
push ebx
push ecx
push esi
...
pop esi
pop ecx
pop ebx
```

이 순서가 틀리면 register가 잘못 복구된다.

예를 들어 다음처럼 하면 잘못된 코드이다.

```asm
push ebx
push ecx
push esi
...
pop ebx
pop ecx
pop esi
```

마지막에 push된 것은 `esi`인데, 먼저 `ebx`로 pop해버리면 값이 서로 뒤바뀐다.

---

## 5. 이 페이지의 핵심

```text
Register parameter는 빠를 수 있지만,
register 값 보존 문제가 생긴다.
보존을 위해 push/pop을 하면 성능 장점이 줄어든다.
push/pop 순서를 틀리면 치명적인 버그가 생긴다.
```

---

# 4페이지. Stack Parameters

이 페이지는 register parameter의 대안으로 **stack parameter**를 설명한다.

## 1. Stack parameter란?

Stack parameter는 argument를 register가 아니라 stack에 push해서 procedure에 전달하는 방식이다.

기존 register parameter 방식은 다음과 같았다.

```asm
mov esi, OFFSET array
mov ecx, LENGTHOF array
mov ebx, TYPE array
call DumpMem
```

이를 stack parameter 방식으로 바꾸면 다음과 같다.

```asm
push TYPE array
push LENGTHOF array
push OFFSET array
call DumpMem
```

이 방식은 특정 register를 parameter 전달용으로 강제하지 않는다.

---

## 2. Stack parameter의 장점

```text
1. 특정 register에 의존하지 않는다.
2. parameter 개수가 많아도 stack에 계속 push하면 된다.
3. procedure 내부에서 EBP 기준으로 parameter를 일관되게 접근할 수 있다.
4. C/C++ 함수 호출 구조와 잘 맞는다.
```

---

## 3. Argument의 두 종류

강의자료는 stack에 push되는 argument를 두 종류로 나눈다.

```text
1. Value arguments
2. Reference arguments
```

---

## 4. Value argument

Value argument는 변수나 상수의 **값 자체**를 넘기는 방식이다.

```asm
push 5
push val1
```

이 경우 procedure는 값의 복사본을 받는다.

C/C++로 보면 다음과 같다.

```c
func(x);
```

함수 내부에서 x 값을 바꿔도 caller의 원본 변수는 바뀌지 않는다.

---

## 5. Reference argument

Reference argument는 값이 아니라 **주소**를 넘기는 방식이다.

```asm
push OFFSET array
push OFFSET val1
```

이 경우 procedure는 원본 데이터의 위치를 알 수 있다.

C/C++로 보면 다음과 같다.

```c
func(&x);
func(array);
```

procedure는 주소를 따라가서 실제 메모리를 읽거나 수정할 수 있다.

---

## 6. 이 페이지의 핵심

```text
Stack parameter는 argument를 stack에 push해서 전달하는 방식이다.
Stack에 push되는 argument는 값일 수도 있고 주소일 수도 있다.
value argument는 복사본 전달이다.
reference argument는 원본 위치 전달이다.
```

---

# 5페이지. Passing by Value

이 페이지는 **passing by value**, 즉 값에 의한 전달을 설명한다.

## 1. 기본 예시

```asm
.data
val1 DWORD 5
val2 DWORD 6

.code
push val2
push val1
call AddTwo
```

여기서 stack에 push되는 것은 `val1`, `val2`의 주소가 아니다.

실제 값인 5와 6이 push된다.

```text
stack top 근처
──────────────
5       ← val1의 복사본
6       ← val2의 복사본
──────────────
```

정확한 push 순서상 `push val2` 후 `push val1`이므로, `val1`이 stack top에 더 가깝다.

---

## 2. C++ equivalent

슬라이드에서는 다음 C++ 호출과 같다고 설명한다.

```c
int sum = AddTwo(val1, val2);
```

이때 `val1`, `val2`는 값으로 전달된다.

함수 내부에서 parameter를 바꿔도 원래 변수는 바뀌지 않는다.

```c
void f(int x)
{
    x = 100;
}

int a = 5;
f(a);
// a는 여전히 5
```

assembly에서도 마찬가지이다. stack에 올라간 것은 원본이 아니라 복사본이기 때문이다.

---

## 3. Reverse order

C/C++ standard convention에서는 arguments가 reverse order로 push된다.

함수 호출이 다음과 같다면,

```c
AddTwo(val1, val2);
```

assembly에서는 보통 다음과 같이 오른쪽 argument부터 push한다.

```asm
push val2
push val1
call AddTwo
```

이렇게 하면 procedure 안에서 첫 번째 parameter가 `[EBP+8]`, 두 번째 parameter가 `[EBP+12]`에 놓인다.

---

## 4. 이 페이지의 핵심

```text
Passing by value는 값의 복사본을 stack에 push한다.
procedure 내부에서 parameter를 바꿔도 원본 변수는 바뀌지 않는다.
C/C++ convention에서는 arguments를 reverse order로 push한다.
```

---

# 6페이지. Passing by Reference

이 페이지는 **passing by reference**, 즉 주소에 의한 전달을 설명한다.

## 1. 기본 예시

```asm
.data
val1 DWORD 5
val2 DWORD 6

.code
push OFFSET val2
push OFFSET val1
call Swap
```

여기서 stack에 push되는 것은 5와 6이 아니다.

```text
offset(val1)
offset(val2)
```

즉, 변수의 주소가 전달된다.

---

## 2. C++ equivalent

슬라이드에서는 다음과 같은 의미로 보여준다.

```c
Swap(&val1, &val2);
```

즉 `Swap`은 val1과 val2의 주소를 받는다.

C 코드로 표현하면 다음과 비슷하다.

```c
void Swap(int* a, int* b)
{
    int temp = *a;
    *a = *b;
    *b = temp;
}
```

---

## 3. Assembly에서 reference parameter 사용

procedure 안에서는 parameter로 받은 주소를 register에 넣는다.

```asm
mov esi, [ebp+8]     ; val1의 주소
mov edi, [ebp+12]    ; val2의 주소
```

그 다음 dereference해서 실제 값을 읽는다.

```asm
mov eax, [esi]       ; val1의 실제 값 읽기
mov ebx, [edi]       ; val2의 실제 값 읽기
```

그리고 다시 원본 메모리에 쓸 수 있다.

```asm
mov [esi], ebx       ; val1 = val2
mov [edi], eax       ; val2 = old val1
```

---

## 4. Passing by value와 reference 비교

```text
Passing by value
    stack에 값의 복사본 push
    procedure가 원본을 바꾸지 못함

Passing by reference
    stack에 원본의 주소 push
    procedure가 원본을 직접 바꿀 수 있음
```

---

## 5. 이 페이지의 핵심

```text
Passing by reference는 주소를 전달한다.
procedure는 그 주소를 이용해 원본 변수에 직접 접근한다.
C의 pointer 또는 call by reference 느낌과 같다.
```

---

# 7페이지. Passing Arrays

이 페이지는 배열 전달 방식을 설명한다.

## 1. 배열은 보통 reference로 전달된다

고급 언어에서는 배열을 함수에 넘길 때 배열 전체를 복사하지 않는다.

배열의 시작 주소만 넘긴다.

```asm
.data
array DWORD 50 DUP(?)

.code
push OFFSET array
call ArrayFill
```

여기서 `OFFSET array`는 배열의 첫 번째 원소 주소이다.

---

## 2. 왜 배열을 value로 전달하지 않는가?

예를 들어 다음 배열이 있다고 하자.

```asm
array DWORD 50 DUP(?)
```

DWORD는 4바이트이고 원소가 50개이므로 전체 크기는 200바이트이다.

이 배열을 value로 전달하려면 원소 50개를 전부 stack에 push해야 한다.

```text
array[0] push
array[1] push
array[2] push
...
array[49] push
```

이 방식은 다음 문제가 있다.

```text
1. 너무 느리다.
2. stack 공간을 많이 사용한다.
3. 함수 호출 비용이 커진다.
4. 배열이 클수록 비효율이 심각해진다.
```

그래서 배열은 거의 항상 주소만 전달한다.

---

## 3. Procedure 내부에서 배열 접근

배열 주소를 받으면 procedure는 그 주소를 pointer처럼 사용한다.

```asm
mov esi, [ebp+8]      ; array 시작 주소
mov ax, [esi]         ; 첫 번째 WORD 또는 DWORD 일부 읽기
add esi, TYPE WORD    ; 다음 원소로 이동
```

즉, 배열 자체가 넘어간 것이 아니라 배열의 시작 위치가 넘어간다.

---

## 4. 이 페이지의 핵심

```text
배열은 보통 reference로 전달된다.
즉, 배열 전체가 아니라 배열 시작 주소가 stack에 push된다.
배열을 value로 전달하면 원소 전체를 복사해야 하므로 매우 비효율적이다.
```

---

# 8페이지. Accessing Stack Parameters

이 페이지는 stack parameter를 procedure 내부에서 어떻게 접근하는지 설명한다.

## 1. Procedure prologue

일반적인 procedure는 다음 코드로 시작한다.

```asm
AddTwo PROC
    push ebp
    mov  ebp, esp
```

이 두 줄은 stack frame을 만들기 위한 기본 코드이다.

```asm
push ebp      ; caller의 EBP 저장
mov ebp, esp  ; 현재 ESP를 새로운 기준점으로 설정
```

---

## 2. Stack 구조

caller가 다음과 같이 호출했다고 하자.

```asm
push 6
push 5
call AddTwo
```

`call AddTwo`가 실행되면 return address가 자동으로 push된다.

그 뒤 `AddTwo`가 시작되면서 다음을 실행한다.

```asm
push ebp
mov ebp, esp
```

그러면 stack은 다음과 같은 구조가 된다.

```text
높은 주소
────────────────────
[EBP+12]   6              ; second parameter
[EBP+8]    5              ; first parameter
[EBP+4]    return address
[EBP]      old EBP
────────────────────
낮은 주소
```

---

## 3. Parameter 접근

이제 procedure 내부에서는 EBP 기준 offset을 사용해서 parameter에 접근한다.

```asm
mov eax, [ebp+8]      ; 첫 번째 parameter
add eax, [ebp+12]     ; 두 번째 parameter
```

왜 첫 번째 parameter가 `[EBP+8]`인가?

```text
[EBP]    = old EBP
[EBP+4]  = return address
[EBP+8]  = first parameter
```

따라서 parameter는 `[EBP+8]`부터 시작한다.

---

## 4. Base-offset addressing

이 방식이 바로 **base-offset addressing**이다.

```text
base register = EBP
offset = 8, 12, 16, ...
```

예:

```asm
mov eax, [ebp+8]
```

이 명령어는 다음 의미이다.

```text
EBP 기준으로 8바이트 위쪽 주소에 있는 값을 EAX로 가져와라.
```

---

## 5. Symbolic constants 사용

매번 `[ebp+8]`, `[ebp+12]`를 직접 쓰면 의미를 알기 어렵다.

그래서 symbolic constant를 사용할 수 있다.

```asm
x_param EQU [ebp+8]
y_param EQU [ebp+12]

mov eax, x_param
add eax, y_param
```

이렇게 하면 코드가 훨씬 읽기 좋아진다.

---

## 6. 이 페이지의 핵심

```text
procedure가 시작되면 EBP를 stack frame 기준점으로 설정한다.
parameter는 EBP보다 위쪽에 있다.
첫 번째 parameter는 보통 [EBP+8]이다.
base-offset addressing으로 stack parameter에 접근한다.
```

---

# 9페이지. 32-bit Calling Conventions: C Calling Convention

이 페이지부터는 **calling convention**을 설명한다.

## 1. Calling convention이란?

Calling convention은 procedure 호출에 대한 약속이다.

구체적으로는 다음을 정한다.

```text
1. parameter를 어디에 둘 것인가?
2. parameter를 어떤 순서로 전달할 것인가?
3. return value는 어디에 둘 것인가?
4. stack 정리는 caller가 할 것인가, callee가 할 것인가?
```

---

## 2. C calling convention

C/C++에서 전통적으로 많이 사용하는 방식이다.

특징은 다음과 같다.

```text
1. argument를 stack에 reverse order로 push한다.
2. procedure를 call한다.
3. procedure가 ret으로 돌아온다.
4. caller가 stack을 정리한다.
```

예시는 다음과 같다.

```asm
Example1 PROC
    push 6
    push 5
    call AddTwo
    add  esp, 8      ; remove arguments from the stack
    ret
Example1 ENDP
```

---

## 3. add esp, 8의 의미

`push 6`, `push 5`를 했으므로 stack에는 DWORD parameter 2개가 쌓였다.

DWORD는 4바이트이므로 총 8바이트이다.

```asm
add esp, 8
```

은 stack pointer를 원래 위치로 되돌려 parameter 공간을 제거하는 코드이다.

---

## 4. Caller cleanup

C calling convention의 핵심은 다음이다.

```text
stack cleanup = caller responsibility
```

즉, procedure를 부른 쪽이 stack을 정리한다.

---

## 5. 왜 caller가 정리하는가?

C 언어에는 가변 인자 함수가 있다.

```c
printf("%d %d %d", a, b, c);
printf("%d", a);
```

함수 호출마다 argument 개수가 달라질 수 있다.

callee 입장에서는 정확히 몇 바이트를 정리해야 하는지 알기 어려울 수 있다.

따라서 caller가 자신이 push한 만큼 직접 정리하는 방식이 자연스럽다.

---

## 6. 이 페이지의 핵심

```text
C calling convention에서는 argument가 reverse order로 push된다.
procedure가 return한 뒤 caller가 add esp, n으로 stack을 정리한다.
```

---

# 10페이지. STDCALL Calling Convention

이 페이지는 **STDCALL calling convention**을 설명한다.

## 1. STDCALL의 특징

STDCALL도 argument를 reverse order로 stack에 push한다.

하지만 C calling convention과 다른 점은 stack cleanup 주체이다.

```text
C calling convention:
    caller가 stack cleanup

STDCALL:
    callee가 stack cleanup
```

---

## 2. RET n

STDCALL에서는 procedure 끝에 다음과 같은 명령어를 사용한다.

```asm
ret 8
```

여기서 8은 parameter가 차지한 총 바이트 수이다.

DWORD parameter 2개라면 4 + 4 = 8바이트이다.

---

## 3. RET 8의 동작

일반 `ret`은 return address만 pop해서 caller로 돌아간다.

하지만 `ret 8`은 다음 두 가지를 수행한다.

```text
1. return address를 pop해서 EIP로 이동한다.
2. ESP에 8을 더해서 parameter 공간을 제거한다.
```

즉, callee가 자기 parameter 공간까지 정리하고 돌아간다.

---

## 4. 코드 비교

C calling convention:

```asm
push 6
push 5
call AddTwo
add esp, 8
```

STDCALL:

```asm
push 6
push 5
call AddTwo
```

callee 내부:

```asm
ret 8
```

---

## 5. STDCALL의 장점

강의자료는 다음 장점을 말한다.

```text
1. caller 쪽 코드가 줄어든다.
2. caller가 stack cleanup을 잊어버릴 위험이 없다.
3. Windows API 호출 방식과 잘 맞는다.
```

32-bit Windows API를 호출하려면 argument를 stack에 전달해야 하며, STDCALL convention이 자주 사용된다.

---

## 6. 이 페이지의 핵심

```text
STDCALL은 callee cleanup 방식이다.
callee는 ret n을 사용해 return address 처리와 parameter 제거를 함께 수행한다.
```

---

# 11페이지. Saving and Restoring Registers

이 페이지는 procedure 내부에서 register를 저장하고 복구하는 법을 설명한다.

## 1. 왜 register를 저장하는가?

caller가 특정 register에 중요한 값을 넣어두었을 수 있다.

```asm
mov ecx, 100
call MySub
; caller는 여전히 ECX가 100이기를 기대할 수 있음
```

그런데 `MySub` 내부에서 ECX를 마음대로 바꾸면 caller의 값이 망가질 수 있다.

그래서 procedure가 register를 사용하기 전에 기존 값을 stack에 저장하고, 끝나기 전에 복구한다.

---

## 2. 기본 구조

```asm
MySub PROC
    push ebp
    mov  ebp, esp

    push ecx
    push edx

    mov  eax, [ebp+8]

    pop  edx
    pop  ecx

    pop  ebp
    ret
MySub ENDP
```

---

## 3. Stack 구조

register 저장 후 stack은 다음과 비슷하다.

```text
높은 주소
────────────────────
[EBP+8]    parameter
[EBP+4]    return address
[EBP]      old EBP
[EBP-4]    saved ECX
[EBP-8]    saved EDX
────────────────────
낮은 주소
```

슬라이드 그림에서도 parameter, return address, EBP, saved ECX, saved EDX가 stack frame 안에 놓이는 모습을 보여준다.

---

## 4. 저장과 복구 순서

저장:

```asm
push ecx
push edx
```

복구:

```asm
pop edx
pop ecx
```

반드시 반대 순서로 해야 한다.

---

## 5. 이 페이지의 핵심

```text
procedure가 register를 사용하면서 caller의 값을 보존해야 한다면 push로 저장하고 pop으로 복구한다.
push/pop은 반드시 LIFO 순서를 지켜야 한다.
```

---

# 12페이지. Local Variables

이 페이지는 local variable이 stack에 어떻게 만들어지는지 설명한다.

## 1. C 코드 예시

슬라이드의 C 코드는 다음과 같은 구조이다.

```c
void MySub()
{
    int X = 10;
    int Y = 20;
}
```

`X`, `Y`는 local variable이다.

local variable의 특징은 다음과 같다.

```text
1. subroutine 내부에서 생성된다.
2. subroutine 내부에서 사용된다.
3. subroutine이 끝나면 사라진다.
```

---

## 2. Assembly에서 local variable 만들기

```asm
MySub PROC
    push ebp
    mov  ebp, esp
    sub  esp, 8              ; create locals

    mov DWORD PTR [ebp-4], 10 ; X
    mov DWORD PTR [ebp-8], 20 ; Y

    mov esp, ebp             ; remove locals from stack
    pop ebp
    ret
MySub ENDP
```

---

## 3. sub esp, 8의 의미

`int X`, `int Y`는 각각 4바이트이다.

```text
X = 4 bytes
Y = 4 bytes
총 8 bytes
```

그래서 stack에 8바이트 공간을 예약한다.

```asm
sub esp, 8
```

---

## 4. Local variable 위치

local variable은 EBP보다 아래쪽에 있다.

```text
[EBP-4]   X
[EBP-8]   Y
```

그래서 다음 코드가 가능하다.

```asm
mov DWORD PTR [ebp-4], 10
mov DWORD PTR [ebp-8], 20
```

---

## 5. local variable 제거

procedure가 끝나기 전에 다음을 실행한다.

```asm
mov esp, ebp
```

이 명령어는 ESP를 EBP 위치로 되돌린다.

그 결과 `[EBP-4]`, `[EBP-8]`에 있던 local variable 공간은 더 이상 유효하지 않게 된다.

즉, local variable을 한 번에 해제하는 효과가 있다.

---

## 6. 이 페이지의 핵심

```text
local variable은 stack에 생성된다.
parameter는 [EBP+offset], local variable은 [EBP-offset]으로 접근한다.
procedure 종료 시 mov esp, ebp로 local variable 공간을 제거한다.
```

---

# 13페이지. Reference Parameters: ArrayFill Example

이 페이지는 reference parameter를 실제 예제인 `ArrayFill`로 설명한다.

## 1. 예제 목적

`ArrayFill` procedure는 배열을 pseudorandom 16-bit integer로 채운다.

전달되는 argument는 두 개이다.

```text
1. 배열 시작 주소    → reference parameter
2. 배열 길이         → value parameter
```

---

## 2. 호출부

```asm
.data
count = 100
array WORD count DUP(?)

.code
push OFFSET array
push count
call ArrayFill
```

여기서 `OFFSET array`는 배열의 시작 주소이다.

`count`는 배열 길이 값이다.

---

## 3. Procedure 내부 parameter 접근

슬라이드에서는 다음과 같이 접근한다.

```asm
mov esi, [ebp+12]     ; points to the array
mov ecx, [ebp+8]      ; array length
```

왜 array가 `[ebp+12]`이고 count가 `[ebp+8]`인가?

호출부에서 push 순서를 보면:

```asm
push OFFSET array
push count
call ArrayFill
```

나중에 push한 `count`가 stack top에 더 가까워진다. 따라서 첫 번째로 접근되는 parameter 위치 `[EBP+8]`에는 count가 있다. 그 위쪽 `[EBP+12]`에는 array 주소가 있다.

---

## 4. Procedure 흐름

개념적으로 코드는 다음과 같다.

```asm
ArrayFill PROC
    push ebp
    mov  ebp, esp
    pushad                    ; save registers

    mov esi, [ebp+12]          ; array 시작 주소
    mov ecx, [ebp+8]           ; array length
    cmp ecx, 0
    je  L2

L1:
    mov eax, 10000h
    call RandomRange           ; random value 생성
    mov [esi], ax              ; 현재 array element에 저장
    add esi, TYPE WORD         ; 다음 WORD 원소로 이동
    loop L1

L2:
    popad                      ; restore registers
    pop ebp
    ret 8                      ; clean up stack
ArrayFill ENDP
```

---

## 5. `mov [esi], ax`의 의미

`ESI`에는 배열의 현재 원소 주소가 들어 있다.

```asm
mov [esi], ax
```

은 AX 값을 ESI가 가리키는 메모리 위치에 저장한다는 뜻이다.

즉, C로 보면 다음과 비슷하다.

```c
array[i] = randomValue;
```

---

## 6. `add esi, TYPE WORD`의 의미

배열은 WORD 배열이다.

WORD는 2바이트이다.

따라서 다음 원소로 이동하려면 주소를 2 증가시켜야 한다.

```asm
add esi, TYPE WORD
```

`TYPE WORD`는 2이므로 다음과 같다.

```asm
add esi, 2
```

---

## 7. `ret 8`의 의미

parameter가 두 개이고 각각 stack에 DWORD 크기로 전달되었다고 보면 총 8바이트이다.

STDCALL convention에서는 callee가 stack을 정리한다.

```asm
ret 8
```

은 return하면서 parameter 공간 8바이트도 제거한다.

---

## 8. 이 페이지의 핵심

```text
reference parameter는 주소를 전달한다.
ArrayFill은 array 주소와 count 값을 받는다.
array 주소는 ESI에 넣고, count는 ECX에 넣어 loop를 돈다.
WORD 배열이므로 다음 원소로 갈 때 ESI를 2 증가시킨다.
```

---

# 14페이지. LEA Instruction

이 페이지는 **LEA, Load Effective Address** 명령어를 설명한다.

## 1. LEA의 역할

LEA는 effective address를 계산해서 register에 넣는 명령어이다.

```asm
lea esi, [ebp-30]
```

이 코드는 `[ebp-30]` 위치에 있는 값을 읽는 것이 아니다.

`ebp - 30`이라는 주소 자체를 계산해서 ESI에 넣는다.

---

## 2. MOV와 LEA 차이

```asm
mov esi, [ebp-30]
```

이 코드는 `[ebp-30]` 주소에 저장된 **값**을 ESI에 넣는다.

반면,

```asm
lea esi, [ebp-30]
```

이 코드는 `[ebp-30]`의 **주소**를 ESI에 넣는다.

비유하면 다음과 같다.

```text
mov esi, [ebp-30]
    그 집 안에 있는 물건을 가져온다.

lea esi, [ebp-30]
    그 집 주소를 가져온다.
```

---

## 3. Local array 예시

슬라이드는 다음 C 코드와 비슷한 예제를 보여준다.

```c
void makeArray()
{
    char myString[30];
    for (int i = 0; i < 30; i++)
        myString[i] = '*';
}
```

assembly 흐름은 다음과 같다.

```asm
makeArray PROC
    push ebp
    mov  ebp, esp
    sub  esp, 32             ; myString local array 공간 확보

    lea  esi, [ebp-30]        ; myString 시작 주소 계산
    mov  ecx, 30              ; loop count

L1:
    mov BYTE PTR [esi], '*'   ; 현재 위치에 '*' 저장
    inc esi                   ; 다음 byte로 이동
    loop L1

    add esp, 32               ; local array 제거
    pop ebp
    ret
makeArray ENDP
```

---

## 4. 왜 sub esp, 32인가?

`myString[30]`은 30바이트이다.

그런데 stack alignment를 위해 32바이트를 예약한다.

```text
30 bytes 필요
→ doubleword boundary 정렬을 위해 32 bytes 예약
```

32는 4의 배수이다.

---

## 5. OFFSET과 LEA의 차이

`OFFSET`은 assembly time에 알 수 있는 주소에 사용한다.

```asm
mov esi, OFFSET array
```

전역 배열 `array`의 주소는 assemble time에 계산할 수 있다.

하지만 local variable 주소는 EBP 값이 runtime에 정해져야 계산할 수 있다.

따라서 다음 코드는 assemble되지 않는다.

```asm
mov esi, OFFSET [ebp-30]   ; error
```

이럴 때 사용해야 하는 것이 LEA이다.

```asm
lea esi, [ebp-30]
```

---

## 6. 이 페이지의 핵심

```text
LEA는 값을 읽는 명령어가 아니라 주소를 계산하는 명령어이다.
OFFSET은 compile/assembly time 주소에 사용한다.
LEA는 runtime에 계산되는 effective address에 사용할 수 있다.
```

---

# 15페이지. ENTER Instruction

이 페이지는 **ENTER instruction**을 설명한다.

## 1. ENTER의 역할

ENTER는 procedure의 stack frame을 자동으로 생성하는 명령어이다.

보통 procedure 시작 부분에서 직접 쓰는 코드는 다음과 같다.

```asm
push ebp
mov  ebp, esp
sub  esp, 8
```

이를 ENTER로 줄이면 다음과 같다.

```asm
enter 8, 0
```

---

## 2. ENTER가 수행하는 작업

```asm
enter numbytes, nestlevel
```

ENTER는 대략 다음 작업을 수행한다.

```asm
push ebp
mov  ebp, esp
sub  esp, numbytes
```

즉,

```asm
enter 8, 0
```

은 다음과 비슷하다.

```asm
push ebp
mov  ebp, esp
sub  esp, 8
```

---

## 3. 첫 번째 operand: local variable 공간

```asm
enter 8, 0
```

여기서 `8`은 local variable을 위해 예약할 stack 공간의 크기이다.

예를 들어 DWORD local variable 2개가 필요하면 8바이트이다.

---

## 4. 4바이트 정렬

강의자료는 첫 번째 operand가 doubleword boundary를 맞추기 위해 4의 배수로 round up된다고 설명한다.

```text
6 bytes 필요 → 8 bytes 예약
30 bytes 필요 → 32 bytes 예약
```

---

## 5. 두 번째 operand: lexical nesting level

```asm
enter numbytes, nestlevel
```

두 번째 operand는 lexical nesting level을 의미한다.

일반적인 수업 예제에서는 대부분 `0`을 사용한다.

```asm
enter 8, 0
```

현재 범위에서는 “ENTER가 stack frame 생성 코드를 자동화한다” 정도가 핵심이다.

---

## 6. 이 페이지의 핵심

```text
ENTER는 push ebp, mov ebp, esp, sub esp, local_size를 자동화한다.
첫 번째 operand는 local variable 공간 크기이다.
두 번째 operand는 lexical nesting level이며, 보통 0을 사용한다.
```

---

# 16페이지. LEAVE Instruction

이 페이지는 **LEAVE instruction**을 설명한다.

## 1. LEAVE의 역할

LEAVE는 procedure의 stack frame을 종료한다.

일반적으로 procedure 끝에서 다음과 같이 쓴다.

```asm
mov esp, ebp
pop ebp
ret
```

이를 LEAVE로 줄이면 다음과 같다.

```asm
leave
ret
```

---

## 2. LEAVE가 수행하는 작업

```asm
leave
```

는 대략 다음과 같다.

```asm
mov esp, ebp
pop ebp
```

즉,

```text
1. ESP를 EBP로 되돌려 local variable 공간을 제거한다.
2. old EBP를 pop해서 caller의 EBP를 복구한다.
```

그 다음 `ret`이 return address를 꺼내 caller로 돌아간다.

---

## 3. ENTER/LEAVE 사용 예

직접 작성:

```asm
MySub PROC
    push ebp
    mov  ebp, esp
    sub  esp, 8

    ; procedure body

    mov  esp, ebp
    pop  ebp
    ret
MySub ENDP
```

ENTER/LEAVE 사용:

```asm
MySub PROC
    enter 8, 0

    ; procedure body

    leave
    ret
MySub ENDP
```

---

## 4. 이 페이지의 핵심

```text
LEAVE는 mov esp, ebp + pop ebp를 자동화한다.
ENTER가 stack frame을 만들고, LEAVE가 stack frame을 해제한다.
하지만 시험/추적 관점에서는 직접 풀어쓴 구조를 이해하는 것이 더 중요하다.
```

---

# 17페이지. Recursively Calculating a Sum

이 페이지부터는 **recursion**을 설명한다.

## 1. Recursion이란?

Recursion은 procedure가 자기 자신을 호출하는 구조이다.

```asm
call CalcSum
```

이 `call`이 `CalcSum` 내부에서 다시 실행되면 recursive call이 된다.

---

## 2. 예제 목표

`CalcSum` procedure는 1부터 n까지의 합을 계산한다.

```text
CalcSum(5) = 5 + 4 + 3 + 2 + 1 = 15
```

슬라이드에서는 다음 register를 사용한다.

```text
ECX = counter, 즉 n
EAX = sum, 즉 누적 합
```

main의 흐름은 다음과 같다.

```asm
mov ecx, 5       ; count = 5
mov eax, 0       ; hold the sum
call CalcSum     ; calculate sum
```

---

## 3. CalcSum procedure

개념적으로 코드는 다음과 같다.

```asm
CalcSum PROC
    cmp ecx, 0       ; counter value 확인
    jz  L2           ; ECX == 0이면 종료

    add eax, ecx     ; EAX += ECX
    dec ecx          ; ECX--
    call CalcSum     ; recursive call

L2:
    ret
CalcSum ENDP
```

---

## 4. 실행 과정

초기 상태:

```text
ECX = 5
EAX = 0
```

1번째 호출:

```text
EAX = 0 + 5 = 5
ECX = 4
```

2번째 호출:

```text
EAX = 5 + 4 = 9
ECX = 3
```

3번째 호출:

```text
EAX = 9 + 3 = 12
ECX = 2
```

4번째 호출:

```text
EAX = 12 + 2 = 14
ECX = 1
```

5번째 호출:

```text
EAX = 14 + 1 = 15
ECX = 0
```

ECX가 0이 되면 더 이상 recursive call을 하지 않고 `ret`으로 빠져나간다.

---

## 5. Stack에 return address가 쌓임

재귀 호출에서 중요한 점은 `CALL`이 실행될 때마다 return address가 stack에 push된다는 것이다.

처음 main에서 CalcSum을 호출하면 return address는 main의 다음 지점이다.

```asm
call CalcSum
L1:
call WriteDec
```

그래서 처음에는 L1이 stack에 push된다.

그런데 CalcSum 내부에서 자기 자신을 호출하면, 돌아갈 주소는 CalcSum 내부의 다음 지점이다.

```asm
call CalcSum
L2:
ret
```

그래서 그 이후에는 L2가 반복해서 stack에 쌓인다.

---

## 6. 이 페이지의 핵심

```text
Recursion은 procedure가 자기 자신을 호출하는 구조이다.
CALL이 반복될 때마다 return address가 stack에 쌓인다.
base case가 없으면 recursion은 끝나지 않는다.
CalcSum에서는 ECX == 0이 base case이다.
```

---

# 18페이지. Calculating a Factorial

이 페이지는 recursion을 이용해 factorial을 계산하는 예제이다.

## 1. Factorial 정의

```text
5! = 5 × 4 × 3 × 2 × 1 = 120
```

수학적으로는 다음과 같다.

```text
factorial(0) = 1
factorial(n) = n × factorial(n-1)
```

C 코드로 보면 다음과 같다.

```c
int factorial(int n)
{
    if (n == 0)
        return 1;
    else
        return n * factorial(n - 1);
}
```

---

## 2. main에서 호출

```asm
main PROC
    push 5
    call Factorial
    call WriteDec
    call Crlf
    exit
main ENDP
```

여기서는 `n = 5`를 stack parameter로 전달한다.

```asm
push 5
call Factorial
```

---

## 3. Factorial procedure 구조

```asm
Factorial PROC
    push ebp
    mov  ebp, esp

    mov eax, [ebp+8]    ; n 가져오기
    cmp eax, 0
    ja  L1              ; n > 0이면 recursive case

    mov eax, 1          ; n == 0이면 1 반환
    jmp L2

L1:
    dec eax             ; n - 1
    push eax            ; factorial(n-1)의 parameter
    call Factorial

ReturnFact:
    mov ebx, [ebp+8]    ; 현재 frame의 n
    mul ebx             ; EAX = EAX * EBX

L2:
    pop ebp
    ret 4
Factorial ENDP
```

---

## 4. `[EBP+8]`이 n인 이유

Factorial이 호출되면 stack에는 다음이 들어간다.

```text
[EBP+8]   n
[EBP+4]   return address
[EBP]     old EBP
```

따라서 다음 명령어는 parameter n을 EAX로 가져온다.

```asm
mov eax, [ebp+8]
```

---

## 5. Base case

```asm
cmp eax, 0
ja  L1
mov eax, 1
jmp L2
```

의미는 다음과 같다.

```text
n > 0이면 L1으로 가서 recursive call 수행
n == 0이면 EAX = 1 반환
```

factorial의 base case는 `0! = 1`이다.

---

## 6. Recursive case

```asm
L1:
    dec eax
    push eax
    call Factorial
```

현재 n이 5라면:

```text
EAX = 5
dec eax → 4
push eax → Factorial(4)의 parameter
call Factorial
```

즉, `Factorial(5)` 안에서 `Factorial(4)`를 호출한다.

---

## 7. 돌아오면서 곱셈

recursive call이 끝나면 EAX에는 `Factorial(n-1)`의 결과가 들어 있다.

그 다음:

```asm
mov ebx, [ebp+8]
mul ebx
```

을 수행한다.

현재 frame의 n을 EBX에 가져오고, EAX와 곱한다.

```text
EAX = Factorial(n-1)
EBX = n
EAX = EAX * EBX = n * Factorial(n-1)
```

---

## 8. 12!까지만 가능한 이유

EAX는 32-bit register이다.

unsigned 32-bit 최대값은 다음과 같다.

```text
2^32 - 1 = 4,294,967,295
```

그런데:

```text
12! = 479,001,600
13! = 6,227,020,800
```

13!은 32-bit 범위를 넘는다.

따라서 EAX 하나로 안전하게 저장 가능한 가장 큰 factorial은 12!이다.

---

## 9. 이 페이지의 핵심

```text
Factorial은 stack parameter로 n을 전달한다.
procedure 내부에서는 [EBP+8]로 n을 가져온다.
n == 0이면 1을 반환한다.
n > 0이면 Factorial(n-1)을 호출하고, 돌아온 결과에 n을 곱한다.
```

---

# 19페이지. Factorial Stack Diagram

이 페이지는 factorial recursion에서 stack frame이 쌓이고 풀리는 과정을 그림으로 보여준다.

## 1. Recursive call마다 stack frame이 새로 생김

`Factorial(5)`가 `Factorial(4)`를 호출하고, `Factorial(4)`가 `Factorial(3)`을 호출한다.

그러면 같은 procedure 코드가 실행되지만 stack frame은 매번 새로 생긴다.

```text
Factorial(5) frame
Factorial(4) frame
Factorial(3) frame
Factorial(2) frame
Factorial(1) frame
Factorial(0) frame
```

각 frame은 독립적인 `[EBP+8]` 값을 가진다.

```text
Factorial(5)의 [EBP+8] = 5
Factorial(4)의 [EBP+8] = 4
Factorial(3)의 [EBP+8] = 3
Factorial(2)의 [EBP+8] = 2
Factorial(1)의 [EBP+8] = 1
Factorial(0)의 [EBP+8] = 0
```

---

## 2. 가장 안쪽 호출부터 반환됨

stack은 LIFO 구조이다.

따라서 가장 마지막에 호출된 `Factorial(0)`이 먼저 끝난다.

```text
Factorial(0)
→ Factorial(1)
→ Factorial(2)
→ Factorial(3)
→ Factorial(4)
→ Factorial(5)
→ main
```

---

## 3. 돌아오면서 곱셈 수행

반환 과정에서 다음 계산이 이루어진다.

```text
Factorial(0) = 1
Factorial(1) = 1 × 1 = 1
Factorial(2) = 1 × 2 = 2
Factorial(3) = 2 × 3 = 6
Factorial(4) = 6 × 4 = 24
Factorial(5) = 24 × 5 = 120
```

---

## 4. 이 페이지의 핵심

```text
재귀 호출은 같은 procedure를 호출하지만, 각 호출마다 독립적인 stack frame이 생긴다.
각 stack frame은 자기만의 parameter n과 return address를 가진다.
재귀 종료 후에는 가장 안쪽 frame부터 LIFO 순서로 반환된다.
```

---

# 20페이지. INVOKE Directive

이 페이지부터는 MASM이 procedure 호출을 편하게 해주는 directive를 설명한다.

## 1. INVOKE란?

`INVOKE`는 procedure 호출을 더 고급스럽고 읽기 쉽게 작성하기 위한 MASM directive이다.

직접 작성하면 다음과 같은 코드를:

```asm
push TYPE array
push LENGTHOF array
push OFFSET array
call DumpArray
```

`INVOKE`를 사용하면 다음처럼 쓸 수 있다.

```asm
INVOKE DumpArray, OFFSET array, LENGTHOF array, TYPE array
```

MASM이 내부적으로 argument push와 call을 생성한다.

---

## 2. INVOKE는 instruction이 아니다

`INVOKE`는 CPU가 직접 실행하는 machine instruction이 아니다.

assembler가 assemble time에 해석해서 실제 `push`, `call` 명령어들로 바꿔주는 directive이다.

```text
source code:
    INVOKE DumpArray, OFFSET array, LENGTHOF array, TYPE array

assembler가 변환:
    push TYPE array
    push LENGTHOF array
    push OFFSET array
    call DumpArray
```

---

## 3. Syntax

일반 형태는 다음과 같다.

```asm
INVOKE procedureName [, argumentList]
```

예:

```asm
INVOKE AddTwo, 5, 6
INVOKE DumpArray, OFFSET array, LENGTHOF array, TYPE array
```

---

## 4. EAX, EDX overwritten 주의

슬라이드에서 중요한 주의사항이 나온다.

작은 크기의 argument, 예를 들어 BYTE나 WORD를 procedure에 전달하면, INVOKE가 이를 stack에 push하기 전에 32-bit 크기로 확장할 수 있다.

이 과정에서 assembler가 EAX나 EDX를 임시로 사용할 수 있고, 그 결과 EAX/EDX가 overwrite될 수 있다.

중요한 값이 EAX나 EDX에 들어 있다면 다음처럼 저장해야 한다.

```asm
push eax
push edx

INVOKE SomeProc, arg1, arg2

pop edx
pop eax
```

---

## 5. 이 페이지의 핵심

```text
INVOKE는 argument push와 procedure call을 자동화한다.
코드를 C 함수 호출처럼 읽기 쉽게 만든다.
하지만 작은 argument를 넘길 때 EAX, EDX가 overwrite될 수 있으므로 주의해야 한다.
```

---

# 21페이지. ADDR Operator

이 페이지는 `ADDR` operator를 설명한다.

## 1. ADDR의 역할

`ADDR`은 `INVOKE`를 사용할 때 주소를 argument로 넘기기 위한 operator이다.

예:

```asm
INVOKE FillArray, ADDR myArray
```

이 코드는 `myArray`의 주소를 `FillArray`에 넘긴다는 뜻이다.

---

## 2. OFFSET과 비슷한 느낌

직접 stack parameter를 쓸 때는 다음처럼 주소를 넘겼다.

```asm
push OFFSET array
call FillArray
```

INVOKE에서는 다음처럼 쓸 수 있다.

```asm
INVOKE FillArray, ADDR array
```

즉, `ADDR`은 INVOKE 안에서 pointer argument를 넘길 때 쓰는 표현이다.

---

## 3. ADDR의 제한

슬라이드는 다음 제한을 말한다.

```text
The argument passed to ADDR must be an assembly time constant.
```

즉, ADDR에 넘기는 대상은 assemble time에 주소가 계산 가능해야 한다.

가능한 예:

```asm
INVOKE FillArray, ADDR myArray
```

불가능한 예:

```asm
INVOKE mySub, ADDR [ebp+12]    ; error
mov esi, ADDR myArray          ; error
```

`[ebp+12]`는 runtime에 EBP 값이 정해져야 주소가 결정된다.

---

## 4. Array element 주소 전달 예

슬라이드는 다음과 같은 예를 보여준다.

```asm
.data
Array DWORD 20 DUP(?)

.code
INVOKE Swap,
    ADDR Array,
    ADDR [Array+4]
```

이것은 내부적으로 다음과 비슷해진다.

```asm
push OFFSET Array+4
push OFFSET Array
call Swap
```

즉, 첫 번째 DWORD 원소와 두 번째 DWORD 원소의 주소를 넘기는 방식이다.

---

## 5. 이 페이지의 핵심

```text
ADDR은 INVOKE에서 주소를 넘기기 위한 operator이다.
ADDR에 전달되는 대상은 assembly-time constant여야 한다.
주소 전달은 passing by reference와 연결된다.
```

---

# 22페이지. PROC Directive

이 페이지는 parameter list를 가진 `PROC` directive를 설명한다.

## 1. 기존 PROC

기존에는 procedure를 다음과 같이 작성했다.

```asm
MyProc PROC
    ...
MyProc ENDP
```

그리고 parameter는 직접 stack offset으로 접근했다.

```asm
mov eax, [ebp+8]
add eax, [ebp+12]
```

문제는 `[ebp+8]`, `[ebp+12]`만 봐서는 무슨 parameter인지 알기 어렵다는 것이다.

---

## 2. Named parameters

MASM의 PROC directive는 parameter에 이름을 붙일 수 있다.

예:

```asm
AddTwo PROC,
    x:DWORD,
    y:DWORD

    mov eax, x
    add eax, y
    ret

AddTwo ENDP
```

이렇게 하면 코드에서 `x`, `y`라는 이름으로 parameter에 접근할 수 있다.

MASM이 내부적으로 `x`, `y`를 `[ebp+8]`, `[ebp+12]` 같은 stack offset으로 처리한다.

---

## 3. 장점

```text
1. stack offset을 직접 계산하지 않아도 된다.
2. 코드 가독성이 좋아진다.
3. parameter 의미가 분명해진다.
4. INVOKE, PROTO와 함께 사용하기 좋다.
```

---

## 4. 기존 방식과 비교

기존 방식:

```asm
AddTwo PROC
    push ebp
    mov  ebp, esp

    mov eax, [ebp+8]
    add eax, [ebp+12]

    pop ebp
    ret 8
AddTwo ENDP
```

PROC parameter list 사용:

```asm
AddTwo PROC,
    x:DWORD,
    y:DWORD

    mov eax, x
    add eax, y
    ret
AddTwo ENDP
```

두 번째 방식이 훨씬 읽기 쉽다.

---

## 5. 이 페이지의 핵심

```text
PROC directive는 procedure에 named parameter list를 붙일 수 있게 해준다.
그 결과 [ebp+8] 같은 stack offset 대신 x, y 같은 이름으로 parameter를 사용할 수 있다.
```

---

# 23페이지. PROC와 RET Instruction

이 페이지는 `PROC` directive와 calling convention이 결합될 때 `RET`이 어떻게 처리되는지 설명한다.

## 1. STDCALL과 PROC

`.MODEL`에서 STDCALL이 기본 protocol이라고 하자.

```asm
.model flat, stdcall
```

이 상태에서 parameter가 있는 procedure를 PROC로 선언하면, MASM은 parameter 크기를 알고 있다.

예를 들어:

```asm
AddTwo PROC,
    x:DWORD,
    y:DWORD
```

parameter는 DWORD 2개이다.

```text
x = 4 bytes
y = 4 bytes
총 8 bytes
```

STDCALL에서는 callee가 stack을 정리해야 하므로, procedure 끝에서는 `ret 8`이 필요하다.

MASM은 PROC의 parameter 정보를 이용해 적절한 entry/exit code 또는 RET 처리를 생성할 수 있다.

---

## 2. Calling convention override

PROC directive의 attributes field를 사용하면 parameter passing protocol을 지정할 수 있다.

즉, `.MODEL`에서 지정한 기본 convention을 procedure별로 override할 수 있다.

개념적으로는 다음과 같은 형태이다.

```asm
MyProc PROC C,
    x:DWORD,
    y:DWORD
```

또는:

```asm
MyProc PROC STDCALL,
    x:DWORD,
    y:DWORD
```

핵심은 `PROC`가 단순히 procedure 이름만 선언하는 것이 아니라, parameter list와 language convention 정보를 assembler에게 줄 수 있다는 것이다.

---

## 3. C convention과 STDCALL 다시 비교

C convention:

```asm
push arg2
push arg1
call MyProc
add esp, 8
```

caller가 cleanup한다.

STDCALL:

```asm
push arg2
push arg1
call MyProc
```

callee가 `ret 8`로 cleanup한다.

---

## 4. 이 페이지의 핵심

```text
PROC는 parameter list와 calling convention 정보를 assembler에게 알려준다.
STDCALL에서는 parameter 크기에 따라 RET n이 필요하다.
PROC의 attributes field는 기본 calling convention을 procedure 단위로 override할 수 있다.
```

---

# 24페이지. PROTO Directive

이 페이지는 `PROTO` directive를 설명한다.

## 1. PROTO란?

`PROTO`는 procedure prototype을 만드는 directive이다.

C/C++의 함수 원형 선언과 비슷하다.

C에서:

```c
int AddTwo(int x, int y);
```

MASM에서:

```asm
AddTwo PROTO,
    x:DWORD,
    y:DWORD
```

---

## 2. 왜 PROTO가 필요한가?

MASM에서 `INVOKE`를 사용하려면 assembler가 procedure 정보를 알고 있어야 한다.

```asm
INVOKE AddTwo, 5, 6
```

이 코드를 올바르게 변환하려면 MASM은 다음을 알아야 한다.

```text
1. AddTwo가 존재하는 procedure인가?
2. parameter가 몇 개인가?
3. 각 parameter 크기는 무엇인가?
4. 어떤 calling convention을 사용하는가?
```

이를 알려주는 것이 PROTO이다.

---

## 3. PROTO는 INVOKE보다 먼저 나와야 함

강의자료는 다음을 강조한다.

```text
MASM requires a prototype for each procedure called by INVOKE.
PROTO must appear first before INVOKE.
```

즉, INVOKE로 procedure를 호출하기 전에 PROTO가 먼저 있어야 한다.

---

## 4. 예시

```asm
AddTwo PROTO,
    x:DWORD,
    y:DWORD

.code
main PROC
    INVOKE AddTwo, 5, 6
    exit
main ENDP

AddTwo PROC,
    x:DWORD,
    y:DWORD

    mov eax, x
    add eax, y
    ret

AddTwo ENDP
```

흐름은 다음과 같다.

```text
1. PROTO로 AddTwo의 형태를 미리 선언한다.
2. main에서 INVOKE AddTwo를 사용한다.
3. 아래쪽에서 AddTwo PROC로 실제 procedure를 정의한다.
```

---

## 5. 이 페이지의 핵심

```text
PROTO는 procedure의 이름과 parameter list를 미리 선언한다.
INVOKE를 사용하려면 procedure prototype이 필요하다.
PROTO는 INVOKE보다 먼저 나와야 한다.
```

---

# 25페이지. PROTO Directive 정리

마지막 페이지는 PROTO의 핵심을 반복한다.

## 1. PROTO의 역할

```text
PROTO는 기존 procedure에 대한 prototype을 생성한다.
prototype은 procedure 이름과 parameter list를 선언한다.
procedure를 정의하기 전에 호출할 수 있게 해준다.
```

---

## 2. PROC, PROTO, INVOKE 관계

세 directive의 관계는 다음과 같다.

```text
PROTO
    procedure의 사전 선언

PROC
    procedure의 실제 정의

INVOKE
    PROTO/PROC 정보를 이용해 argument push + call 자동화
```

C/C++와 비교하면 다음과 같다.

```c
// prototype
int AddTwo(int x, int y);

// call
AddTwo(5, 6);

// definition
int AddTwo(int x, int y)
{
    return x + y;
}
```

MASM에서는 다음과 같다.

```asm
; prototype
AddTwo PROTO,
    x:DWORD,
    y:DWORD

; call
INVOKE AddTwo, 5, 6

; definition
AddTwo PROC,
    x:DWORD,
    y:DWORD

    mov eax, x
    add eax, y
    ret

AddTwo ENDP
```

# 전체 핵심 정리

## 1. Stack frame

```text
procedure 호출 시 stack에 만들어지는 작업 공간이다.
passed arguments, return address, local variables, saved registers가 포함된다.
```

## 2. EBP 기준 접근

```text
[EBP+8]    first parameter
[EBP+12]   second parameter
[EBP-4]    first local variable
[EBP-8]    second local variable
```

## 3. Calling convention

```text
C calling convention:
    caller가 stack cleanup
    call 후 add esp, n

STDCALL:
    callee가 stack cleanup
    ret n 사용
```

## 4. Value vs Reference

```text
Passing by value:
    값의 복사본 전달

Passing by reference:
    주소 전달
    원본 수정 가능
```

## 5. LEA

```text
MOV은 값을 가져오고,
LEA는 주소를 계산해서 가져온다.
```

## 6. Recursion

```text
recursive call마다 새로운 stack frame이 생긴다.
CALL마다 return address가 stack에 쌓인다.
base case가 있어야 recursion이 종료된다.
```

## 7. INVOKE / ADDR / PROC / PROTO

```text
INVOKE:
    argument push + call 자동화

ADDR:
    INVOKE에서 주소 전달

PROC:
    named parameter를 가진 procedure 정의

PROTO:
    procedure prototype 선언
```

---

# 시험 대비 암기 포인트

```text
1. 첫 번째 stack parameter는 보통 [EBP+8]에 있다.
2. [EBP+4]에는 return address가 있다.
3. [EBP]에는 old EBP가 있다.
4. local variable은 [EBP-4], [EBP-8]처럼 음수 offset으로 접근한다.
5. C calling convention은 caller cleanup이다.
6. STDCALL은 callee cleanup이며 ret n을 사용한다.
7. passing by value는 값 복사, passing by reference는 주소 전달이다.
8. 배열은 거의 항상 reference로 전달한다.
9. LEA는 값을 읽는 것이 아니라 effective address를 계산한다.
10. recursion은 호출마다 독립적인 stack frame을 만든다.
11. INVOKE를 쓰면 MASM이 push와 call을 생성한다.
12. PROTO는 INVOKE보다 먼저 필요하다.
```
