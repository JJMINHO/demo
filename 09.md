## Slide 1. Memory Management

첫 슬라이드는 이번 챕터의 제목 슬라이드다.

이번 장의 제목은 **Memory Management**이다. 여기서 말하는 memory management는 두 층위에서 이해해야 한다.

예를 들어 C에서 `malloc`, C++에서 `new`, Java에서 객체 생성이 일어날 때 필요한 메모리를 runtime system 또는 heap manager가 잡아주는 과정을 말한다.

두 번째는 **CPU/OS 수준의 메모리 관리**이다.  
프로그램이 보는 주소가 실제 RAM 주소와 바로 같은 것이 아니라, x86 구조에서는 segment, descriptor, page table, page translation 같은 과정을 거쳐 실제 physical address로 변환된다.

```text

```

## Slide 2. Dynamic Memory Allocation 기본 개념

이 슬라이드의 핵심은 **dynamic memory allocation**, 즉 **heap allocation**이다.

### 1. Dynamic memory allocation이란?

Dynamic memory allocation은 프로그램이 실행되는 도중에 필요한 메모리를 요청하는 기법이다.

정적 메모리 할당과 비교하면 다음과 같다.

```text
정적 할당
- 프로그램을 assembly/compile할 때 크기가 정해짐
- 예: .data 영역의 배열, 전역 변수
- 실행 중 크기 변경이 어렵다

동적 할당
- 프로그램 실행 중 필요한 크기를 요청함
- 예: C의 malloc, C++의 new, Java의 객체 생성
- 실행 중 상황에 맞게 메모리 크기를 유연하게 조절 가능
```

예를 들어 배열 크기를 사용자가 입력해야 알 수 있다면, `.data`에 고정 크기 배열을 선언하는 것보다 실행 중 필요한 만큼 heap에서 할당받는 방식이 자연스럽다.

### 2. Heap manager의 역할

C, C++, Java 같은 언어에는 보통 runtime heap manager가 있다.

Heap manager는 프로그램이 다음과 같은 요청을 할 때 중간에서 처리한다.

```text
메모리 주세요     → allocation
메모리 다 썼어요  → deallocation
```

Heap manager는 보통 프로그램 시작 시 OS로부터 큰 메모리 덩어리를 받아둔다. 그리고 그 내부를 여러 block으로 나누어 관리한다.

### 3. Free list

슬라이드에 나오는 중요한 표현이 **free list**이다.

Free list는 현재 비어 있는 memory block들의 목록이다.

```text
Heap 전체
[사용 중][빈 블록][사용 중][빈 블록][빈 블록][사용 중]
           ↑                    ↑
        free list가 이런 빈 공간들을 추적함
```

프로그램이 메모리를 요청하면 heap manager는 free list에서 충분히 큰 block을 찾는다. 찾으면 그 block을 reserved 상태로 표시하고, 그 block의 시작 주소를 pointer로 반환한다.

### 4. Allocation 과정

Allocation은 대략 다음 흐름이다.

```text
1. 프로그램이 N byte 요청
2. heap manager가 free list 탐색
3. N byte 이상 들어갈 수 있는 빈 block 찾기
4. 해당 block을 reserved/allocated 상태로 표시
5. block의 시작 주소를 pointer로 반환
```

여기서 반환되는 pointer는 이후 메모리를 읽고 쓰는 기준점이 된다.

### 5. Deallocation 과정

Deallocation은 사용이 끝난 block을 다시 free list로 돌려놓는 작업이다.

```text
1. 프로그램이 pointer를 넘기며 이 메모리를 반환하겠다고 요청
2. heap manager가 해당 block을 더 이상 사용 중이 아니라고 표시
3. free list에 다시 연결
4. 이후 다른 allocation 요청에서 재사용 가능
```

### 6. 시험적으로 잡아야 하는 포인트

이 슬라이드에서 반드시 가져가야 하는 말은 다음이다.

```text
Dynamic memory allocation은 실행 중에 heap에서 메모리를 요청하는 방식이고,
heap manager는 free list를 이용해 사용 가능한 block을 추적한다.
할당 시에는 충분히 큰 block을 찾아 pointer를 반환하고,
해제 시에는 block을 free list로 되돌린다.
```

## Slide 3. Assembly에서의 Dynamic Memory Allocation

이 슬라이드는 일반적인 heap 개념에서 assembly program으로 넘어간다.

고급 언어에서는 `malloc`, `new`, 객체 생성 같은 문법으로 heap allocation을 사용한다. 하지만 assembly에서는 그런 고급 문법이 없으므로, 운영체제가 제공하는 API를 직접 호출해야 한다.

이 강의자료에서는 Windows 환경을 기준으로 설명한다.

```text
Assembly program
   ↓ Win32 API 호출
Windows OS
   ↓ heap 관리
Memory block 반환
```

슬라이드의 핵심 문장은 다음이다.

```text
You can request multiple blocks of memory of varying sizes from Windows,
using several Win32 API functions.
```

즉, assembly에서도 Windows API를 호출하면 다양한 크기의 memory block을 여러 개 요청할 수 있다.

중요한 점은 assembly에서 동적 메모리 할당을 한다는 것이 단순히 `mov`나 `add`만 쓰는 일이 아니라는 것이다. 이제부터는 OS가 제공하는 함수를 `INVOKE`로 호출하고, 반환값이 EAX에 들어오는 구조를 이해해야 한다.

## Slide 4. GetProcessHeap, HeapCreate, HeapDestroy

이 슬라이드는 Windows heap 관련 API 중에서 heap 자체를 얻거나 만들고 제거하는 함수들을 설명한다.

---

### 1. GetProcessHeap

`GetProcessHeap`은 현재 프로그램이 이미 가지고 있는 process heap의 handle을 반환한다.

```asm
INVOKE GetProcessHeap
```

호출 결과는 `EAX`에 들어온다.

```text
성공: EAX = process heap handle
실패: EAX = NULL
```

여기서 handle은 직접 데이터가 들어 있는 메모리 주소라기보다는, Windows가 내부적으로 관리하는 heap 객체를 가리키는 식별자라고 이해하면 된다.

즉, 이후 `HeapAlloc`, `HeapFree`를 호출할 때 다음처럼 이 handle을 넘긴다.

```asm
INVOKE HeapAlloc, hHeap, HEAP_ZERO_MEMORY, ARRAY_SIZE
INVOKE HeapFree,  hHeap, 0, pArray
```

---

### 2. HeapCreate

`HeapCreate`는 현재 program을 위한 새로운 private heap을 만든다.

```asm
HeapCreate PROTO,
    flOptions:DWORD,       ; heap allocation options
    dwInitialSize:DWORD,   ; initial heap size, in bytes
    dwMaximumSize:DWORD    ; maximum heap size, in bytes
```

호출 결과는 역시 `EAX`에 들어간다.

```text
성공: EAX = 새 private heap의 handle
실패: EAX = NULL
```

기본 `flOptions`는 `NULL`이다.

`dwInitialSize`는 처음에 확보할 heap 크기이고, `dwMaximumSize`는 heap이 커질 수 있는 최대 크기이다.

슬라이드에서 말하는 중요한 특징은 다음이다.

```text
initial heap size와 maximum heap size는 page boundary에 맞게 올림 처리된다.
```

보통 page 크기는 4KB 단위로 생각한다. 따라서 1000 byte를 요청하더라도 내부적으로는 page 단위 경계에 맞춰 관리될 수 있다.

또한 `HeapAlloc` 호출이 initial heap size를 넘으면 heap은 grow할 수 있다. 단, maximum heap size를 넘으면 더 이상 grow하지 못하고 allocation 실패가 발생할 수 있다.

---

### 3. HeapDestroy

`HeapDestroy`는 `HeapCreate`로 만든 private heap을 제거한다.

```asm
HeapDestroy PROTO,
    hHeap:DWORD        ; heap handle
```

중요한 점은 `HeapDestroy`는 `HeapCreate`로 만든 private heap에 대해 사용한다는 것이다. 이미 process가 기본으로 가지고 있는 process heap을 함부로 destroy하는 개념이 아니다.

정리하면 다음과 같다.

```text
GetProcessHeap  → 기존 process heap handle 얻기
HeapCreate      → 새로운 private heap 만들기
HeapDestroy     → private heap 제거하기
```

## Slide 5. HeapAlloc, HeapFree, GetLastError

이 슬라이드는 실제 memory block을 할당하고 반환하는 API를 설명한다.

---

### 1. HeapAlloc

`HeapAlloc`은 기존 heap에서 memory block을 하나 할당한다.

```asm
HeapAlloc PROTO,
    hHeap:HANDLE,      ; handle to existing heap block
    dwFlags:DWORD,     ; allocation control flags
    dwBytes:DWORD      ; number of bytes to allocate
```

호출하면 새로 할당된 memory block의 pointer가 `EAX`에 들어간다.

```text
성공: EAX = 새 memory block의 pointer
실패: EAX = NULL
```

여기서 handle과 pointer를 구분해야 한다.

```text
heap handle
- heap 자체를 식별하는 값
- HeapAlloc/HeapFree에 넘겨서 어느 heap에서 할당할지 알려줌

pointer
- 실제로 할당받은 memory block의 시작 주소
- 이 주소를 통해 데이터를 읽고 씀
```

---

### 2. HEAP_ZERO_MEMORY

`dwFlags`에 `HEAP_ZERO_MEMORY`를 넣을 수 있다.

```asm
INVOKE HeapAlloc, hHeap, HEAP_ZERO_MEMORY, ARRAY_SIZE
```

이 옵션을 주면 새로 할당된 memory block이 모두 0으로 초기화된다.

이것은 C에서 `malloc`과 `calloc`을 비교할 때 `calloc`이 0으로 초기화해주는 것과 비슷하게 이해할 수 있다.

---

### 3. HeapFree

`HeapFree`는 이전에 heap에서 할당받은 memory block을 반환한다.

```asm
HeapFree PROTO,
    hHeap:HANDLE,
    dwFlags:DWORD,
    lpMem:DWORD
```

일반적으로 `dwFlags`는 `NULL`, 즉 0으로 둔다.

예시는 다음과 같다.

```asm
INVOKE HeapFree, hHeap, 0, pArray
```

의미는 다음과 같다.

```text
hHeap에서 할당받았던 pArray block을 반환하라.
```

---

### 4. GetLastError

`HeapCreate`, `HeapDestroy`, `GetProcessHeap` 같은 API 호출 중 에러가 발생하면 `GetLastError`를 호출해 더 구체적인 오류 정보를 얻을 수 있다.

Assembly에서는 특히 API가 실패했을 때 `EAX == NULL`인지 확인하는 패턴이 자주 나온다.

---

### 5. 이 슬라이드의 핵심 정리

```text
HeapAlloc은 heap에서 실제 memory block을 할당하고 pointer를 EAX에 반환한다.
HeapFree는 이전에 할당받은 block을 heap으로 반환한다.
HEAP_ZERO_MEMORY 옵션을 주면 할당된 block이 0으로 초기화된다.
API 실패 시에는 EAX가 NULL인지 확인하고, 필요하면 GetLastError를 호출한다.
```

## Slide 6. HeapTest Program 1 — 1000-byte array 생성 후 채우기

이 슬라이드는 실제 assembly program 예제다. 목표는 다음이다.

```text
1. process heap handle을 얻는다.
2. heap에서 1000 byte짜리 배열을 동적으로 할당한다.
3. 배열의 각 byte를 0FFh로 채운다.
4. 배열을 출력한다.
5. 사용이 끝난 배열을 HeapFree로 반환한다.
```

---

### 1. Data section

슬라이드의 data section은 대략 다음 구조다.

```asm
.data
ARRAY_SIZE = 1000
FILL_VAL   EQU 0FFh

hHeap  HANDLE ?      ; handle to the process heap
pArray DWORD  ?      ; pointer to block of memory
newHeap DWORD ?      ; handle to new heap
str1 BYTE "Heap size is: ", 0
```

핵심 변수는 두 개다.

```text
hHeap
- heap handle 저장
- GetProcessHeap이 반환한 EAX 값을 저장함

pArray
- HeapAlloc이 반환한 memory block pointer 저장
- 실제 배열의 시작 주소
```

---

### 2. main procedure 흐름

main procedure는 다음 순서로 진행된다.

```asm
main PROC
    INVOKE GetProcessHeap       ; get handle prog's heap
    .IF eax == NULL             ; if failed, display message
        call WriteWindowsMsg
        jmp quit
    .ELSE
        mov hHeap, eax          ; success
    .ENDIF

    call allocate_array
    jnc arrayOk                 ; failed (CF = 1)?
    call WriteWindowsMsg
    call Crlf
    jmp quit

arrayOk:
    call fill_array
    call display_array
    call Crlf

    ; free the array
    INVOKE HeapFree, hHeap, 0, pArray

quit:
    exit
main ENDP
```

중요한 흐름은 다음과 같다.

```text
GetProcessHeap 성공 → hHeap에 저장
allocate_array 호출 → HeapAlloc으로 1000 byte 확보
CF 확인 → 실패하면 에러 메시지
성공하면 fill_array, display_array 호출
마지막에 HeapFree로 반환
```

여기서 `jnc arrayOk`가 중요하다.

```text
jnc = jump if no carry
CF = 0이면 성공이라고 약속한 것
CF = 1이면 allocation 실패라고 약속한 것
```

즉, 이 program에서는 procedure가 성공/실패를 Carry Flag로 알려준다.

---

### 3. allocate_array procedure

`allocate_array`는 실제 allocation을 수행한다.

```asm
allocate_array PROC USES eax
;
; Dynamically allocates space for the array.
; Receives: EAX = handle to the program heap
; Returns: CF = 0 if the memory allocation succeeds.
;
    INVOKE HeapAlloc, hHeap, HEAP_ZERO_MEMORY, ARRAY_SIZE
    .IF eax == NULL
        stc                 ; return with CF = 1
    .ELSE
        mov pArray, eax     ; save the pointer
        clc                 ; return with CF = 0
    .ENDIF
    ret
allocate_array ENDP
```

핵심은 다음이다.

```text
HeapAlloc 성공 → EAX에 pointer 반환 → pArray에 저장 → CF=0
HeapAlloc 실패 → EAX=NULL → CF=1
```

`stc`는 Carry Flag를 1로 세팅한다.  
`clc`는 Carry Flag를 0으로 클리어한다.

따라서 caller는 `jnc` 같은 조건 jump로 성공/실패를 판단할 수 있다.

---

### 4. fill_array procedure

`fill_array`는 할당된 배열 전체를 하나의 byte 값으로 채운다.

```asm
fill_array PROC USES ecx edx esi
;
; Fills all array positions with a single character.
; Receives: nothing
; Returns: nothing
;
    mov ecx, ARRAY_SIZE         ; loop counter
    mov esi, pArray             ; point to the array

L1:
    mov BYTE PTR [esi], FILL_VAL ; fill each byte
    inc esi                      ; next location
    loop L1

    ret
fill_array ENDP
```

이 부분은 indirect addressing 복습과 연결된다.

```text
ESI = 현재 배열 원소를 가리키는 pointer
[ESI] = ESI가 가리키는 memory 위치의 실제 내용
BYTE PTR [ESI] = 그 위치를 1 byte 크기로 해석
```

반복 구조는 다음과 같다.

```text
ECX = 1000
ESI = pArray

반복:
    [ESI]에 0FFh 저장
    ESI = ESI + 1
    ECX = ECX - 1
    ECX != 0이면 반복
```

즉, 동적으로 할당한 heap memory도 일반 배열처럼 pointer를 이용해 접근할 수 있다.

---

### 5. display_array procedure

`display_array`는 배열의 각 byte를 읽어서 출력한다.

```asm
display_array PROC USES eax ebx ecx esi
;
; Displays the array
; Receives: nothing
; Returns: nothing
;
    mov ecx, ARRAY_SIZE       ; loop counter
    mov esi, pArray           ; point to the array

L1:
    mov al, [esi]             ; get a byte
    mov ebx, TYPE BYTE        ; byte 단위 출력 설정
    call WriteHexB            ; display it
    inc esi                   ; next location
    loop L1

    ret
display_array ENDP
```

읽는 것도 쓰는 것과 구조가 같다.

```text
쓰기: mov BYTE PTR [esi], FILL_VAL
읽기: mov al, [esi]
```

이 예제에서 꼭 기억할 포인트는 다음이다.

```text
HeapAlloc으로 받은 주소는 pArray에 저장되고,
ESI 같은 register에 복사한 뒤 [ESI] 형태로 dereference하여 배열처럼 사용한다.
```

## Slide 7. HeapTest Program 2 — heap 크기를 초과할 때까지 반복 할당

이 예제의 목표는 heap의 한계를 실제로 확인하는 것이다.

슬라이드의 설명은 다음과 같다.

```text
An example that repeatedly allocate large blocks of memory
until the heap size is exceeded
```

즉, 큰 memory block을 반복해서 할당하다가 더 이상 heap에서 할당할 수 없는 순간을 확인한다.

---

### 1. Data section

예제의 핵심 상수는 다음과 같다.

```asm
HEAP_START =  2000000      ; 2 MByte
HEAP_MAX   = 400000000     ; 400 MByte
BLOCK_SIZE =   500000      ; .5 MByte
```

의미는 다음과 같다.

```text
처음 heap 크기: 약 2MB
최대 heap 크기: 약 400MB
한 번에 할당할 block 크기: 약 0.5MB
```

그리고 변수는 다음과 같다.

```asm
hHeap HANDLE ?     ; handle to the heap
pData DWORD  ?     ; pointer to block
```

`hHeap`에는 `HeapCreate`로 만든 private heap handle이 들어간다.  
`pData`에는 매번 `HeapAlloc`으로 받은 memory block pointer가 들어간다.

---

### 2. private heap 생성

main에서는 먼저 private heap을 만든다.

```asm
INVOKE HeapCreate, 0, HEAP_START, HEAP_MAX
.IF eax == NULL
    call WriteWindowsMsg
    call Crlf
    jmp quit
.ELSE
    mov hHeap, eax
.ENDIF
```

흐름은 다음이다.

```text
HeapCreate 호출
성공하면 EAX에 heap handle 반환
hHeap에 저장
실패하면 EAX == NULL
```

이전 예제에서는 `GetProcessHeap`으로 기존 process heap을 얻었다.  
이번 예제에서는 `HeapCreate`로 새 private heap을 만든다는 점이 다르다.

---

### 3. 반복 allocation

핵심 loop는 다음 구조다.

```asm
mov ecx, 2000          ; loop counter

L1:
    call allocate_block ; allocate a block
    .IF Carry?          ; failed?
        mov edx, OFFSET str1
        call WriteString
        jmp quit
    .ELSE
        mov al, '.'
        call WriteChar  ; show progress
    .ENDIF

    ; call free_block    ; enable/disable this line
    loop L1
```

여기서 `ECX = 2000`이다.  
한 번에 0.5MB를 할당하므로 2000번이면 이론상 약 1000MB, 즉 약 1GB를 요청하게 된다.

하지만 heap maximum은 400MB로 설정되어 있다. 따라서 어느 순간 allocation이 실패한다.

성공할 때마다 `.`을 출력하는 이유는 진행 상황을 눈으로 확인하기 위해서다.

---

### 4. free_block을 주석 처리하느냐의 의미

슬라이드에 다음 줄이 있다.

```asm
; call free_block        ; enable/disable this line
```

이 줄을 주석 상태로 두면 다음과 같다.

```text
할당만 계속 하고 해제하지 않음
→ heap 사용량이 계속 증가
→ maximum heap size에 도달
→ 어느 순간 HeapAlloc 실패
```

반대로 이 줄을 활성화하면 다음과 같다.

```text
block 하나 할당
바로 free
다음 block 할당
바로 free
→ heap 공간이 계속 재사용됨
→ 실패가 훨씬 늦게 발생하거나 발생하지 않을 수 있음
```

이 예제는 heap allocation과 deallocation의 차이를 체감하게 해준다.

---

### 5. allocate_block procedure

```asm
allocate_block PROC USES ecx
;
; allocate a block and fill with all zeros.
    INVOKE HeapAlloc, hHeap, HEAP_ZERO_MEMORY, BLOCK_SIZE
    .IF eax == NULL
        stc             ; return with CF = 1
    .ELSE
        mov pData, eax  ; save the pointer
        clc             ; return with CF = 0
    .ENDIF
    ret
allocate_block ENDP
```

Slide 6의 `allocate_array`와 구조가 거의 같다.

```text
성공 → pData에 pointer 저장, CF=0
실패 → CF=1
```

---

### 6. free_block procedure

```asm
free_block PROC USES ecx
    INVOKE HeapFree, hHeap, 0, pData
    ret
free_block ENDP
```

할당받은 `pData` block을 `hHeap`으로 반환한다.

---

### 7. HeapDestroy

프로그램이 끝날 때는 private heap을 제거한다.

```asm
quit:
    INVOKE HeapDestroy, hHeap   ; destroy the heap
    .IF eax == NULL             ; failed?
        call WriteWindowsMsg
        call Crlf
    .ENDIF
    exit
```

이 예제의 핵심은 다음이다.

```text
HeapCreate로 private heap을 만들고,
HeapAlloc으로 block을 반복 할당하고,
필요하면 HeapFree로 block을 반환하고,
마지막에는 HeapDestroy로 heap 자체를 제거한다.
```

# Part 2. x86 Memory Management

---

## Slide 8. x86 Memory Management 기본 용어

이제 강의자료의 흐름이 바뀐다.

앞부분은 application program에서 heap을 어떻게 요청하는지였다.  
이제부터는 x86 processor와 OS가 주소 공간을 어떻게 관리하는지 설명한다.

---

### 1. Multitasking

Multitasking은 여러 program이 동시에 실행되는 것처럼 보이게 하는 기능이다.

실제로 CPU core 하나만 놓고 보면 한 순간에 하나의 instruction만 실행한다. 하지만 processor가 시간을 아주 짧게 쪼개 여러 program에게 나누어 주면, 사용자는 여러 program이 동시에 실행되는 것처럼 느낀다.

```text
Program A 실행 조금
Program B 실행 조금
Program C 실행 조금
다시 Program A 실행 조금
...
```

슬라이드는 processor가 실행 중인 모든 program 사이에서 시간을 나눈다고 설명한다.

---

### 2. Segments

Segment는 program이 사용하는 memory 영역이다. code 또는 data를 담을 수 있다.

```text
Code segment   → 실행할 instruction 저장
Data segment   → 전역 변수, 데이터 저장
Stack segment  → stack, local variable, return address 등 저장
```

Segment는 variable-sized area, 즉 크기가 고정 하나로 정해진 것이 아니라 segment마다 다를 수 있다.

---

### 3. Segmentation

Segmentation은 segment들을 서로 분리하는 방식이다.

왜 필요한가?

Multitasking 환경에서는 여러 program이 동시에 메모리에 올라와 있다. 이때 한 program이 다른 program의 data나 code를 마음대로 건드리면 시스템이 불안정해진다.

Segmentation은 각 program의 memory segment를 분리하여 서로 침범하지 못하게 하는 보호 장치 역할을 한다.

```text
Program A segment
Program B segment
Program C segment

A가 B의 영역을 잘못 접근하려 하면 processor/OS가 막음
```

---

### 4. Segment descriptor

Segment descriptor는 하나의 segment를 설명하는 자료구조다.

포함 정보는 다음과 같다.

```text
base address  → segment 시작 주소
access rights → 읽기/쓰기/실행 권한
size limit    → segment 크기 한계
type          → code/data/stack 등
usage         → 사용 방식
```

즉, descriptor는 segment의 신분증 같은 것이다.

---

### 5. Segment selector

Segment selector는 segment register에 저장되는 16-bit 값이다.

```text
CS, DS, SS, ES, FS, GS
```

이 register들이 segment selector를 담는다.

중요한 점은 protected mode에서 segment register가 segment의 실제 시작 주소를 직접 들고 있는 것이 아니라, descriptor table에서 descriptor를 찾기 위한 selector를 들고 있다는 것이다.

---

### 6. Logical address

Logical address는 다음 두 요소의 조합이다.

```text
Logical address = segment selector + 32-bit offset
```

예를 들어 다음처럼 이해할 수 있다.

```text
DS selector + offset 200h
```

이 말은 “DS가 가리키는 segment 안에서 200h만큼 떨어진 위치”라는 뜻이다.

---

## Slide 9. Logical Address를 Linear Address로 변환

이 슬라이드는 이번 장의 핵심 중 하나다.

x86 protected mode에서 program이 사용하는 주소는 바로 physical memory address가 아니다. 먼저 logical address가 linear address로 바뀐다.

---

### 1. 왜 변환이 필요한가?

슬라이드의 예시는 다음과 같다.

```text
세 개의 program이 있고,
각 program마다 offset 200h 위치에 변수가 있다고 하자.
```

각 program 입장에서는 자기 data segment 안의 `200h` 위치를 사용한다.

하지만 실제 메모리에서 세 변수는 서로 다른 곳에 있어야 한다.

```text
Program A의 offset 200h
Program B의 offset 200h
Program C의 offset 200h
```

세 offset 값은 같지만, 각각 속한 segment가 다르므로 최종 주소는 달라진다.

---

### 2. 1단계: Segment + Offset → Linear Address

첫 번째 단계는 segment 값과 offset을 결합하여 linear address를 만드는 것이다.

슬라이드의 공식은 다음이다.

```text
32-bit linear address = segment base address + 32-bit offset
```

여기서 segment base address는 segment descriptor 안에 들어 있다.

---

### 3. Descriptor table을 통한 변환

흐름은 다음과 같다.

```text
Logical address
= segment selector + offset

segment selector
→ descriptor table에서 segment descriptor 선택

segment descriptor
→ segment base address 제공

linear address
= segment base address + offset
```

즉, selector 자체가 base address는 아니다. selector는 descriptor table 안에서 어떤 descriptor를 볼지 알려주는 index에 가깝다.

---

### 4. Paging이 없으면?

슬라이드는 이 linear address가 physical address일 수도 있다고 말한다.

```text
This linear address could be the variable’s physical address.
```

즉, paging을 사용하지 않는다면 linear address가 곧 physical address처럼 사용될 수 있다.

하지만 Windows, Linux 같은 현대 OS는 paging을 사용한다.

따라서 보통은 다음 두 단계가 모두 필요하다.

```text
1단계: logical address → linear address
2단계: linear address → physical address
```

---

## Slide 10. Paging

이 슬라이드는 paging의 목적과 동작 방식을 설명한다.

---

### 1. Paging을 쓰는 이유

Paging을 사용하면 실제 물리 메모리보다 더 큰 프로그램 조합을 실행할 수 있다.

핵심 아이디어는 다음이다.

```text
프로그램 전체를 RAM에 한 번에 올릴 필요는 없다.
지금 당장 필요한 부분만 RAM에 올리고,
나머지는 disk에 둘 수 있다.
```

예를 들어 1GB짜리 프로그램이 있다고 해서 1GB 전체가 항상 RAM에 있어야 하는 것은 아니다. 현재 실행 중인 code와 접근 중인 data page만 RAM에 있으면 된다.

---

### 2. Page

Program이 사용하는 memory는 작은 단위로 나뉜다. 이 단위가 page이다.

슬라이드에서는 page가 보통 4KB라고 설명한다.

```text
1 page = typically 4KB
```

4KB page라면 page 내부 위치를 표현하는 데 12 bit가 필요하다.

```text
2^12 = 4096 byte = 4KB
```

이 내용은 Slide 14, 15의 10/10/12 bit 구조와 바로 연결된다.

---

### 3. Page in / Page out

프로그램 실행 중 processor와 OS는 inactive page를 memory에서 내리고, 지금 필요한 page를 memory로 올릴 수 있다.

```text
inactive page → disk로 내림
required page → disk에서 RAM으로 올림
```

이런 과정을 통해 실제 RAM보다 더 큰 주소 공간을 사용하는 것처럼 보이게 할 수 있다.

---

### 4. Page directory와 Page table

OS는 현재 memory에 있는 page들을 추적하기 위해 page directory와 page table을 유지한다.

```text
Page directory
→ page table들의 위치를 관리

Page table
→ 실제 physical memory의 page frame 위치를 관리
```

이 구조는 32-bit linear address를 physical address로 변환하는 데 쓰인다.

---

### 5. Page translation

Page translation은 linear address를 physical address로 바꾸는 과정이다.

```text
linear address → page directory/page table 참조 → physical address
```

---

### 6. Page fault

만약 요청한 page가 현재 physical memory에 없다면 processor는 program을 잠시 중단시키고 page fault를 발생시킨다.

그 후 OS가 필요한 page를 disk에서 memory로 가져온다. page가 준비되면 program이 다시 실행된다.

Application program 입장에서는 page translation과 page fault가 자동으로 일어나는 것처럼 보인다.

```text
프로그램: 그냥 주소를 사용함
CPU/OS: 뒤에서 page table 보고 변환하고, 필요하면 page fault 처리
```

---

## Slide 11. Paging과 실제 메모리 사용 화면

이 슬라이드는 Windows 작업 관리자 화면을 통해 paging 개념을 직관적으로 보여준다.

슬라이드의 메시지는 다음이다.

```text
By using paging, a computer can run a combination of programs
that would not otherwise fit into memory.
```

즉, OS는 실제 RAM만 단순히 통째로 사용하는 것이 아니라, page 단위로 memory를 관리한다.

작업 관리자에서 보이는 메모리 사용량, commit, cached, paged pool, non-paged pool 같은 항목들은 OS가 memory를 단순 배열처럼 보는 것이 아니라 복잡한 정책으로 관리한다는 점을 보여준다.

시험 관점에서는 세부 UI 수치를 외우기보다 다음 의미를 잡으면 된다.

```text
Paging 덕분에 OS는 여러 process의 memory를 page 단위로 관리하고,
필요한 page만 RAM에 두며,
나머지는 disk와 연계해 관리할 수 있다.
```

## Slide 12. Descriptor Tables — GDT와 LDT

이 슬라이드는 segment descriptor들이 어디에 저장되는지 설명한다.

---

### 1. Global Descriptor Table, GDT

GDT는 시스템 전체에서 사용하는 descriptor table이다.

슬라이드는 다음과 같이 설명한다.

```text
A single GDT is created when the OS switches the processor
into protected mode during boot up.
```

즉, OS가 부팅 중 processor를 protected mode로 전환할 때 GDT가 만들어진다.

GDT에는 OS 전체에서 공통적으로 필요한 segment descriptor들이 들어간다.

---

### 2. Local Descriptor Table, LDT

LDT는 task 또는 program별로 할당될 수 있는 descriptor table이다.

```text
각 task/program은 자기만의 segment descriptor table을 가질 수 있다.
이것이 LDT이다.
```

---

### 3. LDTR register

LDTR register는 현재 program의 LDT 주소를 담는다.

```text
LDTR → 현재 task의 LDT 위치
```

따라서 CPU는 segment selector를 이용해 적절한 descriptor table에서 segment descriptor를 찾을 수 있다.

---

### 4. 슬라이드 그림 해석

그림은 다음 흐름을 보여준다.

```text
Logical address
= segment selector + offset

segment selector
→ local descriptor table의 특정 descriptor 선택

descriptor
→ segment base address 제공

base address + offset
→ linear address
```

그림에는 SS, DS, CS 같은 segment register와 ESP/EIP/offset이 함께 나온다.

의미는 다음과 같다.

```text
SS: stack segment 선택
DS: data segment 선택
CS: code segment 선택
```

각 segment register의 selector가 LDT의 descriptor를 고르고, descriptor 안의 base address와 offset이 더해져 linear address가 만들어진다.

---

## Slide 13. Segment Descriptor Details

이 슬라이드는 segment descriptor 내부에 어떤 정보가 들어 있는지 정리한다.

---

### 1. Base address

Base address는 segment가 linear address space 안에서 시작하는 위치다.

슬라이드는 32-bit integer라고 설명한다.

```text
Base address = segment 시작 위치
```

공식은 다시 다음과 연결된다.

```text
linear address = base address + offset
```

---

### 2. Privilege level

각 segment는 privilege level을 가질 수 있다.

범위는 0부터 3까지다.

```text
0 = 가장 높은 권한, 보통 OS kernel code
3 = 가장 낮은 권한, 보통 user program
```

이것은 흔히 ring protection과 연결된다.

```text
Ring 0 → kernel mode
Ring 3 → user mode
```

이 구조 덕분에 user program이 kernel memory나 중요한 system 영역에 마음대로 접근하지 못한다.

---

### 3. Segment type

Segment type은 해당 segment가 어떤 종류인지 나타낸다.

```text
Data segment
- read-only 또는 read/write
- 위로 grow하거나 아래로 grow할 수 있음
- stack segment도 data segment 범주에 포함됨

Code segment
- execute-only 또는 execute/read-only
```

Stack은 보통 아래 방향으로 grow하는 구조와 연결된다. 즉, `push`가 일어날 때 ESP가 감소하는 구조가 떠오르면 된다.

---

### 4. Segment present flag

Segment present flag는 해당 segment가 현재 physical memory에 존재하는지를 나타내는 bit이다.

```text
present = 1 → segment가 memory에 있음
present = 0 → 현재 memory에 없음
```

---

### 5. Granularity flag

Granularity flag는 segment limit field를 어떤 단위로 해석할지 결정한다.

```text
G = 0 → limit을 byte 단위로 해석
G = 1 → limit을 4096-byte 단위로 해석
```

4096 byte는 4KB이고, page 크기와도 연결된다.

---

### 6. Segment limit

Segment limit은 segment 크기를 나타내는 20-bit integer이다.

단, Granularity flag에 따라 의미가 달라진다.

```text
G = 0일 때
- limit이 byte 단위
- segment size range: 1 byte ~ 1 MB

G = 1일 때
- limit이 4096-byte 단위
- segment size range: 4 KB ~ 4 GB
```

왜 20-bit limit으로 4GB까지 표현할 수 있는가?

```text
20-bit limit의 최대값 ≈ 2^20 - 1
G = 1이면 단위가 4KB = 2^12 byte
따라서 최대 크기 ≈ 2^20 × 2^12 = 2^32 byte = 4GB
```

이 슬라이드는 시험에 나오기 좋은 암기 포인트가 많다.

```text
Base address
Privilege level 0~3
Segment type
Present flag
Granularity flag
Segment limit
```

## Slide 14. Page Translation 구조

이제 마지막 큰 주제인 page translation이다.

이 슬라이드는 32-bit linear address를 32-bit physical address로 바꾸는 구조를 설명한다.

---

### 1. 사용되는 세 구조

Page translation에는 세 구조가 사용된다.

```text
1. Page directory
2. Page table
3. Page
```

---

### 2. Page directory

Page directory는 page-directory entry들의 배열이다.

슬라이드는 최대 1024개의 32-bit page-directory entry가 있다고 설명한다.

왜 1024개인가?

```text
1024 = 2^10
```

뒤에서 32-bit linear address의 directory field가 10 bit이기 때문이다.

---

### 3. Page table

Page table도 최대 1024개의 32-bit page-table entry를 가진 배열이다.

마찬가지로 table field가 10 bit이므로 1024개 entry 중 하나를 선택할 수 있다.

---

### 4. Page

Page는 실제 주소 공간 단위다.

슬라이드는 page가 4KB 또는 4MB address space라고 설명한다. 이 강의 흐름에서는 일반적인 4KB page를 기준으로 이해하면 된다.

4KB page에서는 page 내부 offset이 12 bit이다.

```text
2^12 = 4096 byte = 4KB
```

---

### 5. 32-bit linear address의 분해

32-bit linear address는 세 field로 나뉜다.

```text
31          22 21          12 11          0
+-------------+--------------+-------------+
| Directory   | Table        | Offset      |
| 10 bits     | 10 bits      | 12 bits     |
+-------------+--------------+-------------+
```

각 field의 역할은 다음과 같다.

```text
Directory field
- page directory entry를 고르는 index

Table field
- page table entry를 고르는 index

Offset field
- 선택된 page frame 안에서 몇 byte 떨어진 위치인지 나타냄
```

---

### 6. CR3 register

CR3 register는 page directory의 시작 주소를 담는다.

```text
CR3 → page directory base address
```

따라서 CPU는 linear address의 directory field를 이용해 CR3가 가리키는 page directory 안에서 적절한 entry를 찾는다.

---

## Slide 15. Page Translation 과정

이 슬라이드는 page translation 과정을 순서대로 정리한다.

---

### 1. 시작점: linear address

먼저 linear address는 linear address space의 특정 위치를 가리킨다.

하지만 paging이 켜져 있다면 이 linear address가 곧바로 physical address는 아니다.

---

### 2. Directory field 10 bit

linear address의 상위 10 bit는 directory field이다.

```text
Directory field → page directory entry index
```

CR3가 page directory의 시작 주소를 담고 있으므로, CPU는 다음처럼 접근한다.

```text
page directory base = CR3
PDE address = CR3 + directory_index × entry_size
```

여기서 PDE는 page-directory entry이다.

PDE 안에는 page table의 base address가 들어 있다.

---

### 3. Table field 10 bit

다음 10 bit는 table field이다.

```text
Table field → page table entry index
```

앞 단계에서 찾은 PDE가 page table의 base address를 알려주므로, CPU는 해당 page table 안에서 table index에 해당하는 PTE를 찾는다.

PTE는 page-table entry이다.

PTE 안에는 실제 physical memory에 있는 page frame의 base address가 들어 있다.

---

### 4. Offset field 12 bit

마지막 12 bit는 offset field이다.

```text
Offset field → page frame 내부 위치
```

PTE에서 얻은 page frame base address에 offset을 더하면 최종 physical address가 된다.

```text
physical address = page frame base address + offset
```

---

### 5. 전체 흐름 한 번에 정리

```text
linear address
= directory field + table field + offset

1. CR3가 page directory 시작 주소를 알려줌
2. directory field로 page directory entry 선택
3. page directory entry가 page table 시작 주소를 알려줌
4. table field로 page table entry 선택
5. page table entry가 physical page frame 시작 주소를 알려줌
6. offset을 더해 정확한 physical address 생성
```

---

### 6. 예시: 0x12345678 분해

32-bit linear address가 다음과 같다고 하자.

```text
linear address = 0x12345678
```

이를 10/10/12 bit로 나누면 다음과 같다.

```text
directory index = 0x048 = 72
page table index = 0x345 = 837
offset = 0x678 = 1656
```

의미는 다음이다.

```text
1. CR3가 가리키는 page directory에서 72번째 entry를 찾음
2. 그 entry가 알려주는 page table에서 837번째 entry를 찾음
3. 그 entry가 알려주는 page frame base에 0x678을 더함
4. 최종 physical address 완성
```

# 전체 핵심 요약

## 1. Dynamic Memory Allocation 핵심

```text
Dynamic memory allocation은 실행 중에 heap에서 필요한 memory block을 요청하는 방식이다.
Heap manager는 free list를 통해 사용 가능한 block을 관리한다.
Assembly에서는 Win32 API를 직접 호출하여 heap allocation을 수행한다.
```

주요 API는 다음과 같다.

```text
GetProcessHeap  → 기존 process heap handle 얻기
HeapCreate      → private heap 생성
HeapAlloc       → heap에서 memory block 할당
HeapFree        → 할당된 memory block 반환
HeapDestroy     → private heap 제거
GetLastError    → API 실패 원인 확인
```

---

## 2. Assembly heap 예제 핵심

```text
HeapAlloc 성공 시 EAX에 pointer가 반환된다.
실패 시 EAX는 NULL이다.
예제에서는 성공/실패를 Carry Flag로 caller에게 알려준다.
동적으로 할당한 memory는 ESI 같은 register에 pointer를 넣고 [ESI]로 접근한다.
```

---

## 3. x86 주소 변환 핵심

x86 protected mode의 주소 변환은 크게 두 단계다.

```text
Logical Address → Linear Address → Physical Address
```

첫 번째 단계는 segmentation이다.

```text
logical address = segment selector + offset
segment selector → descriptor table → segment descriptor
linear address = segment base address + offset
```

두 번째 단계는 paging이다.

```text
linear address → page directory/page table → physical address
```

---

## 4. Page Translation 핵심

32-bit linear address는 다음처럼 나뉜다.

```text
10 bits directory index
10 bits page table index
12 bits page offset
```

따라서 구조는 다음이다.

```text
CR3
→ Page Directory
→ Page Directory Entry
→ Page Table
→ Page Table Entry
→ Page Frame
→ + Offset
→ Physical Address
```
